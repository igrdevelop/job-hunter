## 2) Критичные находки (приоритет P0/P1)

### P0-1. Нет рабочего timeout в `hunter/main.py` при запуске `apply_agent.py`

**Где:** `hunter/main.py` (`_run_apply_agent`)  
**Проблема:**  
В коде есть `except asyncio.TimeoutError`, но `proc.communicate()` не обернут в `asyncio.wait_for(...)`.  
Фактически timeout не срабатывает, зависший подпроцесс может держать lock и останавливать hunt-пайплайн.

**Риск:** остановка автоматического цикла hunt/apply, пропуск вакансий, ручные перезапуски.

**Рекомендация:**  
Добавить явный timeout + kill процесса + лог + Telegram-уведомление (по аналогии с `hunter/telegram_bot.py`).

---

### P0-2. Дублирование и рассинхрон логики трекера (`generate_docs.py` vs `hunter/tracker.py`)

**Где:** `generate_docs.py`, `hunter/tracker.py`  
**Проблема:**  
`generate_docs.py` сам ведет запись в `tracker.xlsx` (собственная логика, стили, дедуп), при этом уже есть централизованный модуль `hunter/tracker.py`.

**Риск:** разный формат и поведение в tracker, скрытые баги дедупликации, сложная поддержка.

**Рекомендация:**  
Свести все операции трекера в один слой (`hunter/tracker.py`), а из `generate_docs.py` вызывать только публичный API модуля трекера.

---

### P1-1. Жестко захардкоженный абсолютный путь в `generate_docs.py`

**Где:** `generate_docs.py` (`TRACKER_PATH = Path("D:/LearningProject/Claude/tracker.xlsx")`)  
**Проблема:** абсолютный путь привязан к одной машине и конкретному каталогу.

**Риск:** неработоспособность на другой среде, CI, другом пользователе или после переезда репозитория.

**Рекомендация:** использовать путь из конфигурации (`hunter.config.TRACKER_PATH`) или вычислять относительно `PROJECT_DIR`.

---

### P1-2. Конфигурация и env загружаются в нескольких местах разными способами

**Где:** `hunter/config.py`, `apply_agent.py`  
**Проблема:**  
`hunter/config.py` использует `dotenv`, а `apply_agent.py` вручную парсит `.env` и дублирует ключи.

**Риск:** drift конфигурации, неожиданные различия поведения режимов CLI/API.

**Рекомендация:** единый слой настроек (например `app/config.py`) и переиспользование в обоих рантаймах.

---

### P1-3. Runtime `sys.path`-хаки в `apply_agent.py`

**Где:** `apply_agent.py` (`sys.path.insert(0, str(PROJECT_DIR))`)  
**Проблема:** обход импортов через изменение `sys.path` в рантайме.

**Риск:** нестабильные импорты, сложность тестирования, побочные эффекты.

**Рекомендация:** привести проект к чистой пакетной структуре и запуску через `python -m ...`.

## 3) Архитектурные и структурные улучшения (P2)

Сейчас рядом лежат рантайм-скрипты, бизнес-логика, пользовательские данные (`Applications/`), документы и утилиты.

```text

    hunter/

    apply/

  data/

    applications/
  tests/
    unit/

```

> Примечание: если полный перенос сейчас дорогой, сделать промежуточный шаг — оставить текущие имена, но выделить хотя бы `src/` и `data/`.

---

### P2-2. Дублирование pre-filter логики между source-модулями и центральным фильтром

**Где:** `hunter/filters.py` + `hunter/sources/*`  
Многие источники делают свой pre-filter, а затем снова работает глобальный фильтр.

**Риск:** расхождение правил между источниками, трудность дебага "почему вакансия пропала".

**Рекомендация:**  
- Source-level filter оставить только для грубого отсечения явного мусора.  
- Доменные бизнес-правила держать централизованно в `hunter/filters.py`.

---

### P2-3. Нет тестового контура

Найдены `0` тестовых файлов (`test_*.py`, `*test*.py`).

**Риск:** регрессии при каждом изменении парсеров/дедупа/трекинга.

**Рекомендация (минимум):**
- Юнит-тесты для URL normalization, dedup key и tracker-статусов.
- Контрактные тесты парсеров источников на сохраненных HTML/JSON fixtures.

## 5) Бэклог задач для агента (готово к исполнению)

Ниже задачи в формате, пригодном для следующего агентного прогона.

### TASK-01 (P0): Fix timeout in hunter apply subprocess

- **Цель:** исключить вечное зависание hunt при запуске `apply_agent.py`.
- **Файлы:** `hunter/main.py`
- **Шаги:**
  1. Обернуть `proc.communicate()` в `asyncio.wait_for(..., timeout=...)`.
  2. При timeout делать `proc.kill()` и `await proc.communicate()`.
  3. Логировать причину + возвращать `False`.
  4. Вынести timeout в config-константу.
- **Проверка:** локально эмулировать долгий subprocess; убедиться, что hunt продолжается.

### TASK-02 (P0/P1): Unify tracker writes

- **Цель:** одна точка правды для `tracker.xlsx`.
- **Файлы:** `generate_docs.py`, `hunter/tracker.py`
- **Шаги:**
  1. Добавить в `hunter/tracker.py` метод(ы) для записи "успешного apply".
  2. В `generate_docs.py` заменить прямую работу с Excel на вызов API трекера.
  3. Удалить дублирующий код форматирования, который не нужен.
- **Проверка:** одна новая строка в tracker при успешной генерации, без дублей.

### TASK-03 (P1): Remove hardcoded paths

- **Цель:** переносимость проекта.
- **Файлы:** `generate_docs.py`, `hunter/config.py`
- **Шаги:**
  1. Все пути получать из конфигурации/`PROJECT_DIR`.
  2. Проверить запуск после копирования проекта в другую папку.
- **Проверка:** нет абсолютных путей к `D:/LearningProject/...` в исходниках.

### TASK-04 (P1): Centralize config loading

- **Цель:** устранить расхождение настроек между модулями.
- **Файлы:** `apply_agent.py`, `hunter/config.py` (или новый `app/config.py`)
- **Шаги:**
  1. Выделить единый конфиг-модуль.
  2. `apply_agent.py` использовать тот же источник, что и `hunter`.
  3. Убрать ручной парсинг `.env` из `apply_agent.py`.
- **Проверка:** значения env одинаково интерпретируются в обоих режимах.

### TASK-05 (P2): Add minimal tests

- **Цель:** снизить риск регрессий.
- **Файлы:** новый `tests/`
- **Шаги:**
  1. Тесты для `normalize_url`, `dedup_key`, `has_successful_entry`-логики.
  2. Тест на поведение fail/skip/success статусов.
- **Проверка:** `pytest` проходит на чистом окружении.
