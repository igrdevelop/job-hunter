# Исследование: почему бот заново обрабатывает уже обработанные вакансии

**Дата:** 2026-06-03
**Источник данных:** экспорт Google-таблицы `Job Tracker (5).xlsx` (лист `Tracker`, 588 строк данных)

---

## TL;DR (короткий ответ)

Дедупликация в боте читает **только локальную базу `tracker.db` (SQLite)**, а Google-таблица —
это **односторонняя зеркальная копия (append-only)**, которую дедуп **никогда не читает обратно**.
Как только локальная `tracker.db` оказывается пустой или «не той» (новое окружение, рестарт
контейнера без сохранения БД, сломанный volume-mount), бот «забывает» всё, что уже лежит в таблице,
и заново обрабатывает живые вакансии — генерирует документы и дописывает новую строку в таблицу.
URL при этом совпадает дословно, потому что сравнивать его не с чем: в локальной БД его нет.

**Доказательство:** из 50 повторных строк **41 (82%)** имеют пустой флаг `Re-application`.
Бот ставит `+` в этой колонке автоматически, если URL уже есть в его БД. Пустой флаг = на момент
записи БД ничего не знала об этом URL → дедуп был «слепой».

### Главная улика — колонка `Folder`

Пути к папкам с документами выдают **минимум три разных окружения**, работающих на **одну общую
Google-таблицу**:

| Префикс пути                                 | Окружение                       | Кол-во строк |
|----------------------------------------------|----------------------------------|--------------|
| `/app/Applications/...`                      | Docker-контейнер                 | 171          |
| `D:/LearningProject/Claude/Applications/...` | Локальная Windows-машина         | 23           |
| `Applications/...` (относительный)           | ещё один запуск (ранний период)  | 288          |
| (пусто)                                      | SKIP / EXPIRED / без документов  | 106          |

Пример одного URL, обработанного в **двух окружениях**:

```
theprotocol.it/.../angular-developer-rzeszow...
  r476  2026-05-26  folder=/app/Applications/2026-05-26/...      (Docker)
  r493  2026-05-28  folder=D:/LearningProject/Claude/App...      (локально)
  r494  2026-05-28  folder=D:/LearningProject/Claude/App...      (локально)
```

Пример одного URL, переобработанного **в том же Docker-окружении два дня подряд и дважды за день**:


## Почему дедуп не срабатывает (механизм)

### 1. Дедуп смотрит только в локальную `tracker.db`

`hunter/main.py` (шаг 3 цикла охоты):

```python
known_urls = await asyncio.to_thread(get_known_urls)   # читает SQLite tracker.db
...
if norm in known_urls or norm in seen_urls_this_run:
    dup_url += 1
    continue
```

`get_known_urls()` (`hunter/tracker.py`):

```python
SELECT url_norm FROM applications WHERE url_norm != ''
```

То есть единственный источник истины для дедупа — таблица `applications` в `tracker.db`.

### 2. Google-таблица — это зеркало, дедуп её не читает

Запись в таблицу идёт только в одну сторону: после успешного `apply` вызывается
`gsheets_sync.mirror_new_row(row)`, который **дописывает** строку. Обратная синхронизация
(`pull_full_snapshot` → `_apply_pull_delta_db`) **сопоставляет строки только по `ID`** и
обновляет поля `Sent / To Learn / Re-application` у **уже существующих** в БД записей.
Она **никогда не вставляет** в БД строки, которых в БД нет:

```python
# hunter/gsheets_sync.py: _apply_pull_delta_db
tracker_rows = {r["ID"]: r for r in read_all_tracker_rows() if r.get("ID")}   # из БД
sheets_by_id = {...}                                                          # из таблицы
for row_id, db_row in tracker_rows.items():        # итерируем по БД, не по таблице!
    if row_id not in sheets_by_id: continue
    ...
```

**Вывод:** если URL есть в таблице, но его нет в локальной `tracker.db`, дедуп его не увидит,
и pull его в БД не вернёт. Таблица не может «напомнить» боту о ранее обработанных вакансиях.

### 3. Когда `tracker.db` оказывается «слепой»

- **Разные окружения.** Docker, локальный Windows и ранний относительный запуск — у каждого
  своя `tracker.db`. Таблица общая, базы разные → кросс-окружные дубли (видно по `Folder`).
- **Рестарт без сохранённой БД.** В `docker-compose.yml` есть явное предупреждение-ловушка:

  ```yaml
  # tracker.db must exist on the host before first run.
  # If it doesn't exist Docker will auto-create a directory, breaking the tracker.
  - ./tracker.db:/app/tracker.db
  ```

  Если `./tracker.db` на хосте не существовал как файл, Docker создаёт на его месте **каталог**,
  mount ломается, и контейнер каждый раз пишет в свежую/пустую БД внутри себя. При
  `restart: always` каждый перезапуск (деплой, краш) даёт пустую `known_urls` →
  переобработка всех ещё живых вакансий. Это объясняет повторы **внутри одного дня**.

### 4. Подтверждение через флаг `Re-application`

`add_applied()` ставит `Re-application = "+"`, если на момент записи URL **уже был** в БД:

```python
is_reapply = bool(norm_url and conn.execute(
    "SELECT 1 FROM applications WHERE url_norm=? LIMIT 1", (norm_url,)).fetchone())
```

Факт по данным:

```
повторные (2-е и далее) строки: 50
  с пустым Re-application (БД была слепой):     41  (82%)
  с "+" (БД помнила -> настоящий повтор):        9  (18%)
```

82% повторов записаны при пустой/слепой БД — это и есть прямое доказательство, что
дедуп на момент записи не знал об уже существующем URL.

## Корневые причины (по убыванию вклада)

1. **Несколько окружений на одну таблицу, но у каждого своя `tracker.db`.**
   Дедуп локальный, таблица общая и только на запись → неизбежные дубли при работе
   с двух машин / из контейнера и локально одновременно.

2. **`tracker.db` не переживает рестарты** (вероятно из-за ловушки с volume-mount —
   путь создан как каталог, либо БД не примонтирована на локальном запуске).
   Каждый старт с пустой БД = переобработка живых вакансий.

3. **Архитектурный разрыв:** Google-таблица — то, что видит пользователь и считает
   «источником истины», но дедуп её не читает. Pull обновляет только существующие по `ID`
   строки и не восстанавливает знание об URL.

## Рекомендации

**Быстрые (убрать симптом):**

- Убедиться, что на хосте `./tracker.db` существует как **файл** до старта контейнера
  (`touch ./tracker.db`), и что он реально примонтирован и переживает рестарты.
- Не запускать бота одновременно из двух окружений на одну таблицу. Выбрать одно
  «боевое» окружение, остальные — только для preview с отдельным `APPLICATIONS_DIR`
  и отдельной/выключенной таблицей.
- Разово почистить таблицу от 50 дублей (можно по `url_norm`, оставляя строку с
  заполненным `Sent` / первую успешную).

**Системные (убрать причину):**

- **Дедуп по URL прямо в Google-таблице.** Перед `apply` проверять не только локальную
  `tracker.db`, но и наличие `url_norm` среди строк таблицы (она уже целиком читается в
  `read_all`). Тогда любое окружение будет видеть общую историю.
- **Bootstrap БД из таблицы при старте.** Расширить `_apply_pull_delta_db` (или добавить
  отдельный шаг), чтобы строки, которые есть в таблице, но отсутствуют в `tracker.db`,
  **вставлялись** в БД (с пересчётом `url_norm`). Тогда свежая/пустая БД самовосстанавливается
  из таблицы и дедуп перестаёт быть «слепым» после рестарта.
- Добавить уникальный индекс/`UNIQUE(url_norm)` или защиту от дублей на стороне зеркала
  таблицы (`mirror_new_row` не дописывает строку, если такой `url_norm` в таблице уже есть).
