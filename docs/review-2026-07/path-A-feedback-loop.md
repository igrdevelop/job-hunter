# Путь A — Замкнуть петлю обратной связи

**Цель:** система перестаёт быть «машиной генерации CV вслепую» и начинает
управляться исходами: какие источники, треки, модели и скоры реально приводят
к ответам работодателей.

**Почему это путь №1:** вся инфраструктура сбора данных уже построена
(funnel, колонка N с вердиктом, confirmation/answer из /check_responses,
dual-apply пары, source_health). Не хватает только шага «посмотреть на данные
и принять решения». Это самый дешёвый путь с самой высокой отдачей — он
определяет, куда тратить усилия во всех остальных путях.

---

## Шаг A1. Аудит источников по воронке (D.3 из июньского ревью)

1. Снять `/funnel 90` (или `compute_funnel(90)` скриптом, если нужен экспорт).
2. Для каждого из 21 источников зафиксировать: tracked / generated / sent /
   confirmed / answered за 90 дней.
3. Разбить на корзины:
   - **Кормильцы** — есть sent и хотя бы один confirmed/answered.
   - **Балласт** — tracked > 0, но sent ≈ 0 (всё фильтруется/дедупится).
   - **Мертвецы** — tracked ≈ 0 за 90 дней (сверить с `/health`: сломан или пуст).
4. Мертвецов выключить тумблерами `*_ENABLED` (код не удалять — это дёшево
   вернуть). Для балласта посмотреть 5–10 отфильтрованных вакансий: фильтр
   прав или источник шлёт мусор?
5. Результат — короткая таблица решений в `docs/review-2026-07/` + коммит
   с изменёнными тумблерами.

**Критерий готовности:** список активных источников осознан; расписание
хантов сжалось (меньше источников = короче 12-часовой цикл).

## Шаг A2. Вердикт по dual-apply: выбрать боевую модель

1. Собрать все пары primary/shadow с начала июля: вердикты обоих лежат в
   content.json (`ats_verdict`), тени — на Drive в
   `{company}/{shadow_profile}/`.
2. Посчитать: средний вердикт primary vs shadow, доля пар где shadow ≥
   primary, стоимость на вакансию каждой стороны (колонка M / llm_cost).
3. Глазами сравнить 10–15 пар (это важнее среднего: вердикт не ловит
   стилистическую деревянность).
4. Принять одно из решений:
   - **Sonnet остаётся** → `/dual off`, экономим теневую генерацию.
   - **DeepSeek достаточно хорош** → сменить профиль (`/llm deepseek-v3`),
     6× экономия; dual оставить ещё на 2 недели уже с Sonnet в роли тени.
   - **Гибрид** — дешёвая модель генерирует, Sonnet только в refine-цикле
     (потребует небольшой доработки llm_profiles: разные профили на разные
     стадии).
5. Зафиксировать решение + цифры в work log CLAUDE.md.

**Критерий готовности:** dual-apply больше не работает «в стол»; выбор модели
подкреплён числами и глазами.

## Шаг A3. Валидация ATS-вердикта против ответов

Вопрос, на который до сих пор нет ответа (июньское ревью, пункт 5 «What could
be better»): предсказывает ли ATS-скор хоть что-нибудь?

1. Дождаться ~30–50 строк, где есть и вердикт (колонка N), и исход
   (confirmation/answer или тишина > 3 недель).
2. Скрипт `tools/verdict_vs_outcome.py` (read-only по tracker.db): разбить
   sent-строки на корзины по вердикту (<85 / 85–92 / 93+), посчитать
   confirm-rate и answer-rate в каждой.
3. Интерпретация:
   - Корреляция есть → цель 95 оправдана, refine-цикл окупается.
   - Корреляции нет → снизить `ATS_VERDICT_TARGET` (например до 85) и/или
     `ATS_VERDICT_MAX_REFINES=1` — каждый лишний раунд это деньги и время;
     вердикт оставить как информационную метрику.
4. То же самое по трекам (Angular vs React vs fullstack): какой трек
   конвертирует — туда и смещать фильтры/усилия.

**Критерий готовности:** у числа «ATS: NN%» появился доказанный (или
опровергнутый) смысл; конфиг refine-цикла подогнан под реальность.

## Шаг A4. Автоматизировать еженедельный срез

Чтобы петля не требовала ручной дисциплины:

1. Новый scheduled-колбэк `scheduled_weekly_funnel` (понедельник 09:00,
   по образцу `scheduled_daily_summary`): `/funnel 7` + `/funnel 30` +
   дельта по источникам → одно Telegram-сообщение.
2. Включить в него: топ-3 источника по sent, число FAIL за неделю,
   средний вердикт за неделю, спенд за неделю (колонка M).
3. Тесты по образцу test_funnel + существующих schedule-тестов.

**Критерий готовности:** раз в неделю в Telegram приходит отчёт, по которому
видно деградацию (источник умер, FAIL растут, спенд пополз) без ручных команд.

---

**Риски пути:** главный — малая выборка (десятки sent, единицы answered);
не делать сильных выводов из <30 наблюдений на корзину, лучше подождать ещё
месяц, чем зафиксировать шум как истину.
