## Шаг B1. Наблюдение за doomed-gate (первые 2 недели после деплоя)

Gate ушёл в master 2026-07-06 с калибровкой 0/394 false positives, но
калибровка ≠ прод.

1. Раз в 2–3 дня: выборка SKIP-строк трекера с момента деплоя, у которых
   причина — gate (HARD-находка в Telegram-сообщении).
2. Каждую проверять глазами: действительно ли вакансия обречена
   (иностранный onsite / W2-C2C / язык)?
3. Ложный SKIP на живой вакансии → немедленно:
   - `DOOMED_GATE_HARD_ACTION=warn` в проде (рычаг для этого и сделан);
   - кейс → регрессионный тест в `test_doomed_gate.py`;
   - фикс паттерна → вернуть `skip`.
4. Параллельно смотреть SOFT-предупреждения: не слишком ли шумят
   (усталость от алертов = их перестают читать).
5. Через 2 недели: короткая запись в work log — сколько HARD-скипов,
   сколько из них верных, сэкономленный спенд (каждый скип ≈ $0.13–0.17).

**Критерий готовности:** есть прод-статистика точности gate; режим `skip`
подтверждён или откачен осознанно.

## Шаг B2. Precision-обзор claim-judge → `JUDGE_MODE=block` (план M4)

Judge работает в `warn` с 2026-06-12 — месяц данных накопился.

1. Собрать `judge_report.json` из папок Applications за июнь–июль
   (скрипт-однодневка или руками — их десятки, не тысячи).
2. Классифицировать fabrication-находки: true positive (реально выдумано) /
   false positive (есть в профиле или постинге, judge проглядел).
3. Если precision fabrication ≈ 100% (ожидаемо — quote-валидация уже
   отрезает галлюцинации судьи) → `JUDGE_MODE=block` в проде. Блок значит
   «сломанный документ не уйдёт», это последний недожатый рубеж.
4. По exaggeration той же выборкой решить вопрос из плана M4: включать ли
   авто-репейр. Если >20% находок спорные (инструмент реально есть в
   профиле) — оставить report-only и подтюнить judge_rules.md, а не код.
5. Тесты: `test_claim_judge.py` уже покрывает block-ветки обоих пайплайнов —
   новых почти не потребуется.

**Критерий готовности:** fabrication блокирует доставку; политика по
exaggeration зафиксирована в CLAUDE.md.

## Шаг B3. Прод-статистика refine-цикла

Цикл (PR #115) дорогой по построению: до 2 рерайтов × (генерация + рендер +
вердикт). Нужно понять его реальную отдачу.

1. По content.json за июль собрать: стартовый вердикт → финальный вердикт,
   сколько раундов принято / откачено keep-best'ом.
2. Ответить на три вопроса:
   - Какой средний прирост даёт раунд 1 (honest)?
   - Как часто раунд 2 (stretch) вообще принимается?
   - Сколько стоит один пункт прироста вердикта?
3. Если раунд 2 почти всегда откатывается — `ATS_VERDICT_MAX_REFINES=1`
   (честный раунд остаётся, stretch выключается, спенд падает).
4. Связать с шагом A3: если вердикт не коррелирует с ответами, targets
   снижаются и этот шаг становится ещё важнее.

**Критерий готовности:** конфиг TARGET/MAX_REFINES выставлен по данным,
а не по умолчанию.
