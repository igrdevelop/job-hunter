# Путь D — Эксплуатация

**Цель:** убрать три известных прод-дыры, которые не решаются кодом из dev
(нужны действия на деплой-хосте), и снизить фон FAIL/429.

**Особенность пути:** почти всё здесь — действия владельца на проде, код
менять почти не нужно. Поэтому шаги короткие, но требуют именно ручного окна.

---

## Шаг D1. LinkedIn-сессия (`LINKEDIN_STORAGE_STATE`) — главный источник FAIL

Без сессии каждый LinkedIn detail-fetch получает 429 → FAIL-строки в трекере
(задокументировано в CLAUDE.md как «the single biggest source of FAIL rows»).

1. На деплой-хосте (или локально, затем скопировать файл):
   `python tools/linkedin_login.py` → логин руками → сохраняется
   storage-state JSON.
2. `.env`: `LINKEDIN_STORAGE_STATE=/path/to/state.json`; в Docker —
   промонтировать файл.
3. Убрать `linkedin.com` из `GMAIL_ENRICH_SKIP_HOSTS` (стабы из писем начнут
   обогащаться полным текстом).
4. Неделю понаблюдать: FAIL-строки по linkedin.com должны почти исчезнуть;
   `/funnel` покажет рост generated по LinkedIn.
5. Учесть: сессия протухает — при возврате 429-волны перелогиниться. Если
   надоест руками, завести алерт по образцу oauth_alert (отдельная мелкая
   задача, только после того как протухание реально случится).

**Критерий готовности:** доля FAIL от LinkedIn в трекере упала на порядок.

## Шаг D2. Пересборка prod-образа (E.1, висит с июня)

Inhire работает только в образе, собранном текущим Dockerfile
(`playwright install chromium --with-deps`).

1. На деплой-хосте: `docker compose build --no-cache && docker compose up -d`
   (образ потяжелеет ~на 500 МБ — проверить место на диске заранее).
2. Проверка: `/hunt inhire` → должен вернуть ненулевой список (в dev
   стабильно 19–25 вакансий).
3. Заодно проверить `/health` после пары циклов: Inhire должен уйти из
   NODATA.

**Критерий готовности:** Inhire даёт вакансии в проде; строка E.1 закрыта
в CLAUDE.md.

## Шаг D3. Включить Gmail-источник (после D1)

`GMAIL_ENABLED=false` по умолчанию. Gmail-алерты — единственный канал,
который видит вакансии из подписок (LinkedIn/NoFluff/JustJoin/Pracuj)
раньше и шире скрейперов, а весь обвес уже построен (per-email отчёт,
enrichment c rate-limiter'ом, skip-hosts).

1. Убедиться, что D1 сделан (иначе LinkedIn-ссылки из писем снова начнут
   стучать 429 — хотя skip-hosts это и гасит).
2. `GMAIL_ENABLED=true` в проде; первый `/hunt gmail` руками, посмотреть
   per-email отчёт.
3. Неделя наблюдения за 429-фоном (`hunter_errors.log` на Drive) —
   лимитер уже настроен, но прод покажет.

**Критерий готовности:** gmail-цикл в расписании, 429-шторма нет, в /funnel
появился источник gmail.

## Шаг D4. Ротация Google OAuth-токенов — проверить алерт боем

`oauth_alert` (E.3) написан и оттестирован, но в проде ещё ни разу не
срабатывал по-настоящему.

1. Ничего не делать заранее. Когда придёт первый Telegram «re-auth needed» —
   проверить, что команда из алерта (`python tools/gsheets_auth.py` и т.п.)
   отрабатывает, и что после реаутентификации не осталось false-EXPIRED
   строк (тот самый исторический каскад).
2. Если алерт молчит, а Sheets перестал обновляться — это баг классификации
   в `is_oauth_error`, завести кейс.

**Критерий готовности:** первый реальный exp/revoke пережит без ручной
раскопки логов.
