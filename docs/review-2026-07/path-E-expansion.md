# Путь E — Расширение (исследовательский)

**Цель:** зафиксировать кандидатов на «следующую большую вещь», отделив
готовое к решению (повторки) от требующего разведки (авто-отправка).
Этот путь стартует **только после** того, как пути A/B дали свои ответы.

---

## Шаг E1. Дедуп повторок («повторка») — единственный созревший пункт

**Проблема** (задокументирована 2026-06-29, ждёт решения владельца):
перепосты вакансий получают новый URL и слегка изменённый title, поэтому
точный дедуп URL+company+title их пропускает → повторная генерация CV на ту
же роль (строки 674, 698, 700, 713, 723, 727, 731, 738, 749, 762 трекера).

Пошагово:

1. **Сначала измерить**: скрипт по tracker.db — сгруппировать строки по
   `company + нормализованный title` (lowercase, без пунктуации/уровней),
   посчитать группы с >1 строкой за последние 90 дней. Это цена проблемы
   в долларах (каждый дубль ≈ $0.15) и штуках.
2. **Решение владельца** по политике (три варианта, нужен выбор):
   - **(a) Блокировать**: нечёткий матч company+title с окном (например
     60 дней) → SKIP с причиной "repost";
   - **(b) Помечать, но пускать**: генерировать, но ставить флаг
     Re-application `+` автоматически (сейчас он ручной) — честно перед
     трекером, деньги тратятся;
   - **(c) Спрашивать**: Telegram-карточка «похоже на повторку строки NNN
     от DD.MM — подать заново?» с кнопками. Самый точный, +1 ручной клик.
3. Реализация (после выбора): нормализатор title (стрип
   Senior/Sr/Lead/(m/f/d)/remote-суффиксов) + `find_similar(company,
   title_norm, days)` в tracker.py-слое; точка встройки — рядом с
   существующим дедупом в `main.py` и `tracker_service.should_skip_url`.
4. Калибровка по образцу doomed-gate: прогнать матчер по историческим
   строкам, убедиться что он находит задокументированные 10 повторок и
   не склеивает разные роли одной компании (у крупных компаний десятки
   разных Angular-вакансий — главный риск false positive).
5. Тесты: позитивные (повторки из аудита) + негативные (разные роли одной
   компании).

**Критерий готовности:** повторка не генерируется молча; политика зафиксирована
в CLAUDE.md.

## Шаг E2. Полуавтоматическая отправка откликов — только разведка

Сейчас пайплайн обрывается на «документы готовы, ссылка в Telegram» —
отправка полностью ручная. Полная автоматизация подачи через формы ATS —
большой и рискованный проект; **не начинать без разведки**:

1. По данным шага A1 посчитать, куда реально уходят sent-отклики:
   доля Workable / Greenhouse / Lever / Recruitee / Ashby (у них стабильные
   формы и уже есть адаптеры в `hunter/ats/`) против «зоопарка» (свои сайты,
   LinkedIn Easy Apply, e-mail).
2. Если ≥30–40% sent приходится на стандартные ATS — сделать прототип на
   ОДНОМ провайдере (Workable как самый простой POST): предзаполнение формы
   данными из candidate_profile + PDF, **отправка только после кнопки
   подтверждения в Telegram** (никакого silent-submit — это репутационный
   риск: один кривой автоответ на живую вакансию хуже десяти неподанных).
3. Если доля <30% — путь закрыть с пометкой «не окупается», как это было
   честно сделано с Wellfound/Jobgether в Queue 3.

**Критерий готовности:** решение go/no-go с цифрами, не ощущениями.

## Шаг E3. Чего в этом пути осознанно НЕТ

- **Новые скрейперы-источники** — подтверждаю июньскую анти-цель; вход
  только через данные (если A1 покажет, что воронка голодает — сначала
  включить Gmail (D3), он дешевле любого нового скрейпера).
- **Web-дашборд** — Telegram + Sheets достаточно для одного пользователя;
  дашборд = новая поверхность поддержки без нового исхода.
- **Мультипользовательность / продуктизация** — другой проект с другими
  требованиями (секреты, изоляция, биллинг); не смешивать с личным
  инструментом.
