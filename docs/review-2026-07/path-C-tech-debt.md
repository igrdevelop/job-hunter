validate_content, все скрабы, language-gate, doomed-gate-раннер.

1. Инвентаризация публичных имён и их импортёров (`grep -r "from hunter.apply_shared import"`).
2. Распил по ответственности, механический перенос без изменения логики:
   - `hunter/apply_scrubs.py` — `_strip_compliance_claims`,
     `_strip_prestige_claims`, `_dedup_skill_glosses` + их хелперы;
   - `hunter/apply_review.py` — CL self-review цикл;
   - `hunter/apply_notify.py` — Telegram-хелперы апплая;
   - в `apply_shared.py` остаются: константы, `ApplyError`,
     `validate_content`, `compute_output_folder`, `enforce_language_separation`
     (гейт трогает обе стороны — ему место в shared), `run_doomed_gate`.
3. В `apply_shared.py` — реэкспорты старых имён на переходный период

4. Тесты не переписывать — они и есть страховка механического переноса;
   только поправить патч-таргеты там, где мокается модуль-владелец.
5. `python -m compileall .` + `ruff check .` + полная суита.

**Критерий готовности:** ни один модуль апплай-слоя не больше ~700 строк;
поведение бит-в-бит (ни один тест не изменил ассерты).

## Шаг C2. mypy-гейт на новые модули (A.3 в урезанном виде)

Полная типизация репо — нереалистично одним заходом. Реалистично — не дать
новому коду родиться нетипизированным.

1. `mypy` в dev-зависимости; в `pyproject.toml` — секция с явным списком
   проверяемых модулей: `verdict_refine.py`, `dual_apply.py`, `funnel.py`,
   `source_health.py`, `lang_guard.py`, `llm_profiles.py` (новые, маленькие,
   почти наверняка уже близки к чистоте).
2. Добить аннотации до зелёного mypy на этом списке.
3. Добавить шаг в CI рядом с ruff.
4. Правило в CLAUDE.md: новый модуль в `hunter/` добавляется в mypy-список.

**Критерий готовности:** CI падает, если новый типизированный модуль ломает
mypy; список расширяется по одному модулю за PR, без кампанейщины.
