# Что сделано хорошо (июль 2026)

Сравнение с июньским ревью (`docs/PROJECT_REVIEW_2026-06.md`): за месяц выполнен
почти весь тогдашний роадмап **и сверх него** — профили LLM, dual-apply,
детерминизация ATS-цикла, независимый PDF-вердикт, refine-цикл, doomed-gate.
Ниже — то, что стоит осознанно сохранять как рабочую культуру проекта.

## 1. Роадмап исполняется, а не лежит

Июньский приоритет C→B→D→A→E закрыт за месяц: claim-judge (C), source-health +
`/health` (B), funnel + `/funnel` (D), ruff на весь репо (A.2), OAuth-алерты
(E.3), актуализация Known Issues (E.2). Это редкая дисциплина: план из ревью
не превратился в декорацию.

## 2. Оборона качества CV — эшелонированная и достроенная до конца

Семь последовательных рубежей, каждый ловит свой класс ошибок:

1. **Prompt RED LINES** (generation_rules.md) — первый, самый дешёвый фильтр.
2. **Sanitizer + детерминированные скрабы** (compliance / prestige / gloss) —
   мгновенные regex-фиксы известных паттернов.
3. **Claim-judge** (Haiku, независимая модель) — системный ловец фабрикаций
   с verbatim-валидацией цитат, нейтрализующей галлюцинации самого судьи.
4. **Language enforce-gate** — единственный рубеж с правом **блокировать
   доставку**: сломанный EN-документ физически не уходит работодателю.
5. **PDF-roundtrip** — скоринг того текста, который реально распарсит ATS,
   а не того, что лежит в content.json.
6. **Независимый вердикт** — модель, которая резюме *не писала*, оценивает
   рендер; только эта цифра видна пользователю (self-score генератора убран
   из UI — правильное решение против самообмана метрикой).
7. **Refine-цикл с keep-best** — регресс невозможен по построению: раунд
   принимается только при строгом улучшении вердикта, иначе полный откат
   content.json + документов.

Ключевая идея — «промптам нельзя верить, нужны детерминированные
пост-контроли» — проведена последовательно через все семь слоёв.

## 3. Оптимизация затрат сделана на данных, а не на ощущениях

Анализ 713 реальных content.json с Drive показал: 88% прогонов сжигали все
5 ATS-рерайтов при keyword=100%, потому что порог 95% был математически
недостижим (комбинированный скор упирался в TF-IDF, который рерайтом не
двигается). Решение — детерминированный ранний выход + перенос LLM-оценки в
пост-рендерный вердикт. Итог: ~8.3 → ~3–4 вызова, ~$0.38 → ~$0.13–0.17 на
вакансию. Плюс prompt-caching системного префикса, effort=low на способных
моделях, и OpenRouter/DeepSeek как 6×-дешёвая альтернатива. Это образцовый
паттерн: сначала померили, потом чинили.

## 4. Doomed-gate: $0-отсев с честной калибровкой

Регексовый (нулевая LLM-стоимость) скрин полного текста вакансии до первого
LLM-вызова, откалиброванный на **394 реальных постингах** с жёсткой планкой
приёмки «0 HARD-находок на вакансиях, которые владелец реально отправил».
Найденные 2 false positives починены по существу (perks-контекст у "onsite",
nice-to-have у языка), а не замазаны; pre-policy случаи (Micro1) честно
выделены отдельной категорией вместо подгонки числа. Есть аварийный рычаг
`DOOMED_GATE_HARD_ACTION=warn` на случай, если прод покажет худшую точность.

## 5. Культура root-cause сохранилась под нагрузкой

Свежие примеры за месяц: тень dual-apply убивала успешный основной апплай
через общий таймаут → вынесена в detached-процесс с собственным watchdog;
шторм 429 от gmail_enricher → skip-hosts вместо глушения симптома; ложные
FAIL-флуды разобраны до стадии fetch с очисткой невиновного PR #107. Фиксы
по-прежнему приходят с защитными инвариантами, а не патчами-симптомами.

## 6. Тесты растут вместе с кодом и остаются быстрыми

1283 → **1706** тестов за месяц, полный прогон **13.8 секунды** — это значит,
что суита реально запускается перед каждым коммитом, а не «когда-нибудь в CI».
Каждый фикс приносит регрессионные тесты; калибровка gate — 4 регрессионных
теста на конкретные прод-кейсы.

## 7. Честность как архитектурное свойство

Stretch-раунд refine-цикла может добавлять технологии из постинга, но: каждое
добавление логируется в To Learn (долг кандидата, видимый в трекере), вплетать
можно только в гибкий Altoros-проект, и **никогда** — в верифицируемых
работодателей (Atruvia, Fairmarkit, Intel, SII, SolbegSoft). Граница между
«подсветить» и «соврать» закреплена в коде и тестах, а не в благих намерениях.

## 8. Операционная устойчивость

Best-effort интеграции (Sheets/Drive падают — пайплайн живёт), dedup
self-heal после потери БД, реконсиляция удалённых строк с защитой от
частичного чтения (`_RECONCILE_MIN_RATIO`), различение «не запушено» от
«удалено пользователем» (`sheets_row IS NOT NULL`), четыре непересекающихся
писателя в Sheet (A–K / L / M / N), OAuth-алерты с кулдауном, суточные бэкапы.

## 9. A/B-инфраструктура для выбора модели

Dual-apply: боевая генерация + теневая (DeepSeek) на одной и той же вакансии,
оба варианта скорит **один и тот же** Anthropic-судья (override не влияет),
тени уходят на Drive для глазного сравнения. Это готовый механизм для
data-driven выбора боевой модели — осталось им воспользоваться (см. путь A).
