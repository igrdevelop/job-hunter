## Порядок выполнения и зависимости

```
Волна 1 (быстрые, независимые):  01, 02, 06
Волна 2 (надёжность + страховка): 03, 04
Волна 3 (большой рефакторинг):    05   ← требует 04 (golden E2E как страховка)
Волна 4 (продукт/развитие):       08, 09  ← 08 легче ПОСЛЕ 05 (меньше мест правки),
                                            09 независим и маленький
Параллельно, дедлайн = публикация: 07
```

Ключевые зависимости:
- **05 не начинать без 04**: рефакторинг ядра без сквозного теста — хождение
  по канату без страховки.
- **08 выгоднее после 05**: после унификации пайплайна персональные константы
  сосредоточатся в меньшем числе модулей.
- **09 не зависит ни от чего** — инфраструктура (base_cv_react.md, запросы
  "react" в источниках, stack-hint детектор) уже существует; меняются только
  фильтры-запреты.

## Принципы (действуют для всех пунктов)

1. **Каждый пункт — отдельная ветка/PR** с тестами; один PR — одна тема
   (правило из memory/git workflow владельца).
2. **Никаких спекулятивных слоёв**: каждый новый механизм должен менять
   реальное решение или ловить реальный класс поломки (стоящее правило
   владельца; пункт 01 — прямое его применение к уже существующему слою).
3. **CLAUDE.md обновляется в том же коммите**, что и поведение/файлы.
4. Best-effort контракт неприкосновенен: Sheets/Drive/Telegram/shadow никогда
   не роняют apply. Пункт 03 добавляет *наблюдаемость* поверх контракта, не
   меняя его.
