# 04 — Coverage в CI + golden E2E тест apply-пайплайна

**Приоритет:** P1 · **Усилие:** 1–2 дня · **Ветка:** `test/coverage-and-golden-e2e`
**Блокирует:** 05 (унификация пайплайнов — без этого теста рефакторинг ядра не начинать)

## Для чего апдейт

Две слепые зоны в текущем (сильном — ~2200 тестов) наборе:

1. **Coverage не измеряется вообще.** Нет pytest-cov ни локально, ни в CI —
   при 62k строк неизвестно, *что именно* покрыто. Вероятные дыры: ветки
   ошибок в `gsheets_sync`, telegram-handlers, редкие пути `apply_cli`.
2. **Нет ни одного сквозного теста пайплайна.** Все тесты — юниты вокруг
   отдельных стадий с моками соседей. Класс багов «стадии по отдельности
   работают, а связка сломана» (переименовали ключ content.json, забыли
   прошить флаг во второй пайплайн, verdict-штамп до создания строки) ловится
   только в проде. Именно этот класс — большинство фиксов в Agent Work Log.

Golden E2E — ещё и обязательная страховка для рефакторинга 05: «поведение
пайплайна не изменилось» должно проверяться одним тестом, а не глазами.

## Как именно будет происходить

### Часть A — coverage (час)

1. `pytest-cov` в dev-зависимости.
2. CI test-job: `pytest tests/ -q --cov=hunter --cov-report=xml --cov-report=term`.
   **Без порога-гейта** на старте (не блокировать деплой) — сначала карта.
3. XML-отчёт скармливается SonarCloud-job'у (см. док 06) — там coverage на
   новый код станет видимым в PR.
4. После 2–3 недель: решить, ставить ли мягкий порог (`--cov-fail-under`) —
   только если цифра стабилизировалась.

### Часть B — golden E2E (основная работа)

Один тест-модуль `tests/test_golden_apply_e2e.py`, который прогоняет
`hunter.apply_api.main_api()` **целиком**, мокая только внешние границы:

- **LLM**: monkeypatch `llm_client.call_llm` → фикстурные ответы
  (генерация, judge, verdict, refine) из `tests/fixtures/golden/`. Судья и
  verdict возвращают детерминированные JSON.
- **Сеть**: `sources.fetch_job_text` → фикстурный `job_posting.txt`
  (реальные постинги уже лежат в `tests/fixtures/sample_jobs/`).
- **LibreOffice/generate_docs**: сабпроцесс-граница — monkeypatch раннера
  команды: создаёт пустые DOCX/PDF-файлы с ожидаемыми именами (сама генерация
  доков покрыта своими тестами; здесь проверяем оркестрацию).
- **Telegram**: мок `_tg_notify`/send-функций, собирающий сообщения в список.
- **БД/файлы**: `APPLICATIONS_DIR` → tmp_path, `tracker.db` → tmp (паттерн
  уже есть в существующих тестах трекера).

Assert-контракт (то, что должно пережить любой рефакторинг):
- `content.json`: полный набор ключей (`resume_en`, `cover_letter_en`,
  `primary_lang`, `ats_verdict`, `cost`, `to_learn`, …);
- строка в tracker.db: company/title/URL/ATS%/Folder/Cost/ats_verdict;
- набор файлов в папке: job_posting.txt, content.json, PDF-имена
  (`Ihar_Petrasheuski_CV_..._ats{NN}.pdf`), outreach.md;
- последовательность Telegram-сообщений (pre-apply ping → docs ready);
- негативные сценарии отдельными кейсами: expired → EXPIRED-строка без LLM;
  doomed-gate HARD → SKIP-строка, $0; language-gate block → нет доков.

## Что меняется в коде

| Файл | Изменение |
|------|-----------|
| `pyproject.toml` | `pytest-cov` в dev extras |
| `.github/workflows/deploy.yml` | флаги `--cov` в test-job; передача xml в sonar-job |
| `sonar-project.properties` | `sonar.python.coverage.reportPaths=coverage.xml` |
| `tests/test_golden_apply_e2e.py` | **Новый**: 4–6 сквозных сценариев (happy EN, happy PL, expired, doomed-HARD, lang-block, paste-mode) |
| `tests/fixtures/golden/` | **Новый**: фикстурные LLM-ответы (JSON) на каждую роль вызова |
| `tests/conftest.py` | Общая фикстура `fake_llm` (роутер по промпту → ответ) — переиспользуема другими тестами |
| Прод-код | **Не меняется** (если для мока границы придётся вынести раннер сабпроцесса в инъектируемую функцию — это единственное допустимое микро-изменение) |

## Критерий готовности

- CI печатает coverage-таблицу; в Sonar видно покрытие.
- Golden-тест падает при намеренной порче связки (напр., закомментировать
  verdict-штамп) — проверить на 2–3 мутациях руками.
- Полный suite остаётся < ~2 минут (фикстуры, никакой сети).

## Риски

- Golden-тест может стать хрупким (падать на каждое легитимное изменение).
  Лечение: ассертить **контракт** (ключи, наличие файлов, статусы строк), а
  не байтовое содержимое текстов.
- Не пытаться покрыть все 25 источников — их контракты покрыты юнитами;
  E2E тестирует оркестрацию, не скрейперы.
