# 07 — Подготовка к публичному репозиторию

**Приоритет:** P1 — жёсткий блокер публикации (сроки диктуются планом split)
**Усилие:** по фазам docs/SCOUT_REPO_SPLIT_PLAN.md + ~полдня на секрет-скан
**Ветка:** по фазам плана split + `ci/gitleaks`

## Для чего апдейт

Репо готовится стать публичным (personal-файлы уже gitignored, tracked только
`.example`-шаблоны). Осталось два риска, оба известны, но не закрыты:

1. **`linkedin_scout/` всё ещё в репо.** LinkedIn-скрейпер со stealth-флагами
   под именем владельца в публичном репо — задокументированный ToS/репутационный
   риск (docs/SCOUT_REPO_SPLIT_PLAN.md). Phase 0 (декаплинг импортов,
   версионированный `/scoutfound`-контракт, golden fixture) сделан 2026-07-08;
   Phases 1–4 (новый приватный репо, десктоп-катовер, чистка основного репо,
   опциональный history scrub) — не начаты.
2. **История git не проверена на секреты.** `.env`, токены, tracker.xlsx или
   личные prompts могли мелькнуть в ранних коммитах до появления .gitignore-
   правил. Публикация раскрывает **всю историю**, не только HEAD.

## Как именно будет происходить

### Часть A — scout split (по существующему плану)

Выполнить Phases 1–4 из `docs/SCOUT_REPO_SPLIT_PLAN.md` как написано:
1. Phase 1: приватный репо `linkedin-scout`, перенос кода + его тестов +
   `docs/LINKEDIN_POSTS_SCOUT_TASK.md`, контрактный тест payload v1 остаётся
   в обоих репо против общей golden-фикстуры.
2. Phase 2: катовер Task Scheduler на владельческом десктопе на новый чекаут.
3. Phase 3: удаление `linkedin_scout/` из основного репо (бот-сторона —
   `commands/scoutfound.py` + `sources/linkedin_scout_relay.py` — остаётся,
   она scrape-free).
4. Phase 4: history scrub — см. часть B, решение становится **обязательным,
   а не опциональным**, если скан что-то найдёт.

### Часть B — секрет-скан (до нажатия «public», после Phase 3)

1. Полный скан истории локально: `gitleaks detect --source . --log-opts="--all"`
   + контрольный `trufflehog git file://.` (два движка — разные сигнатуры).
2. Разбор находок:
   - реальный секрет в истории → **ротация ключа немедленно** (Telegram token,
     Google OAuth secrets, API-ключи) — ротация обязательна даже при
     последующем scrub'е, скрабленная история могла быть уже склонирована;
   - затем `git filter-repo` по путям/строкам + force-push (согласовать с
     владельцем: инвалидация всех клонов/веток);
   - ложные срабатывания → в `.gitleaksignore` с комментарием.
3. Скан личных данных, которых не должно быть публично: телефоны/адреса в
   docs/fixtures (`grep -riE "телефон|phone|\+48" docs/ tests/fixtures/`).
   Имя владельца в CLAUDE.md — ок (публично на CV-сайте), контакты — нет.

### Часть C — gitleaks как постоянный CI-гейт

Job в `deploy.yml` (перед lint): `gitleaks/gitleaks-action@v2` на diff PR —
чтобы секрет не попал в историю **после** чистки.

## Что меняется в коде

| Файл | Изменение |
|------|-----------|
| `linkedin_scout/` | Удаляется из основного репо (Phase 3) |
| `tests/test_linkedin_scout_*.py` | Переезжают в новый репо; контрактный тест payload остаётся |

| `docs/SCOUT_REPO_SPLIT_PLAN.md` | Отметки выполнения фаз |

## Критерий готовности

- `gitleaks detect --log-opts="--all"` по основному репо: 0 находок
  (или все — в .gitleaksignore с обоснованием).
- Scout работает из приватного репо ≥ 1 неделю (relay-поток жив: `/health`
  показывает scout-relay yield).
- CI-джоб gitleaks зелёный и блокирующий.
- Только после всего этого — переключение репо в public.

## Риски

- History scrub (force-push) ломает открытые PR/локальные клоны — делать в
  окно без активных веток, владелец переклонирует.
- После публикации `GHCR`-образ и compose-URL уже светятся в workflow — они
  и рассчитаны на публичность, но `curl docker-compose.yml` с raw.master на
  VPS стоит перепроверить на предмет отсутствия секретов в compose (секреты
  должны оставаться в `.env` на VPS, не в compose).
