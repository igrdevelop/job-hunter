# 01 — Данные по вердикту → решение по ATS_VERDICT_TARGET

**Приоритет:** P0 · **Усилие:** часы · **Ветка:** не нужна (сначала только запуск тулз на проде)

## Для чего апдейт

Refine-loop (`hunter/verdict_refine.py`) срабатывает почти на каждой генерации:
реальные вердикты 72–94, target 95 → 1–3 rewrite-раунда на вакансию (каждый —
полный LLM-вызов + re-render + re-verdict). При этом **нет ни одного
подтверждения**, что вердикт 94 даёт больше ответов работодателей, чем 85.
Реальные ATS-системы (Greenhouse/Lever/Workable/Traffit — те, куда идут наши
отклики) match-процент вообще не считают и по нему не отсеивают; отсеивают
knockout-вопросы и рекрутерский поиск по ключевым словам.

Инструменты для проверки уже написаны (LLM_COST_REDUCTION_PLAN M2), но на
прод-БД **не запускались** — локальная копия tracker.db устарела (нет колонки
`ats_verdict`, проверено 2026-07-15).

Потенциальный эффект: −30–50% LLM-спенда на вакансию при нулевой потере
качества откликов.

## Как именно будет происходить

1. На прод-хосте (или на свежем экспорте прод-`tracker.db`) запустить:
   - `python tools/verdict_funnel_corr.py` — sent/confirmed/answered-rate по
     бэндам вердикта (<80 / 80–84 / 85–89 / 90–94 / 95+);
   - `python tools/verdict_noise.py` — собственный шум судьи (разброс оценки
     одного и того же PDF при повторных прогонах).
2. Матрица решения:
   - **answered-rate бэнда 90–94 ≈ бэнда 80–89** (разница внутри шума судьи)
     → вердикт не предиктивен → снизить `ATS_VERDICT_TARGET` до 88–90
     **или** `ATS_VERDICT_MAX_REFINES` до 1 (одна честная попытка).
   - **шум судьи σ ≥ 3 пунктов** → любые целевые дельты меньше ~2σ считать
     недостижимыми честными раундами; stretch-раунд (добавление технологий)
     оставить только по явному сигналу gap-report, а не по числу.
   - **корреляция есть** (95+ реально отвечают чаще) → target оставить,
     задокументировать вывод, вопрос закрыт.
3. Зафиксировать решение + сырые цифры в `docs/LLM_COST_REDUCTION_PLAN.md`
   (раздел M2) и в Agent Work Log.

## Что меняется в коде

Сам код refine-loop **не трогаем** — меняются только дефолты конфигурации по
результату замера:

| Файл | Изменение |
|------|-----------|
| `hunter/config.py` | Возможное новое значение по умолчанию `ATS_VERDICT_TARGET` (95 → 88–90) и/или `ATS_VERDICT_MAX_REFINES` (3 → 1). По правилу проекта дефолт живёт в config.py и меняется коммитом, не правкой прод-`.env`. |
| `docs/LLM_COST_REDUCTION_PLAN.md` | Раздел M2: результаты замера + принятое решение. |
| `CLAUDE.md` | Таблица Key Configuration: новые дефолты + одна строка «почему» со ссылкой на замер. |

Если данных мало (answered — редкое событие), решение: собрать ещё 3–4 недели
и перезапустить corr-тулзу; до тех пор временная экономия через
`ATS_VERDICT_MAX_REFINES=1` в `.env` как эксперимент (это документированный
механизм override).

## Критерий готовности

Обе тулзы запущены на прод-данных; решение (оставить/снизить) принято и
закоммичено с цифрами; месяц спустя — контрольный прогон `/funnel` +
corr-тулзы: answered-rate не упал.

## Риски

Минимальные: тулзы read-only, no-LLM. Единственный риск — принять решение на
малой выборке; лечится порогом «минимум N=30 строк в сравниваемых бэндах».
